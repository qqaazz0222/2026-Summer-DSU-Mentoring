## 1. 웹 개발 큰 그림

```
[사용자 브라우저]  ←→  [프론트엔드: HTML/CSS/JS]
        │  (HTTP 요청/응답, JSON 데이터)
        ▼
[백엔드 서버: Python/FastAPI]  ←→  [DB: SQLite]
```

- **프론트엔드**: 사용자가 직접 보고 클릭하는 화면. HTML(구조) + CSS(스타일) + JS(동작)
- **백엔드**: 화면 뒤에서 데이터 처리하는 서버. 이 멘토링에선 Python(FastAPI)으로 만듦
- **DB(데이터베이스)**: 데이터를 저장해두는 곳. 서버가 읽고 씀
- **웹 통신**: 프론트와 백엔드가 데이터 주고받는 방식 (HTTP, JSON)

Week1~3은 프론트만, Week4는 통신 개념, Week5~6은 백엔드+DB, Week7~8은 다 합쳐서 프로젝트.

## 2. 클라이언트-서버 & HTTP

| 용어 | 뜻 |
|---|---|
| 클라이언트 (Client) | 요청하는 쪽. 보통 브라우저 |
| 서버 (Server) | 요청받고 응답 주는 쪽. 항상 켜져서 대기 중인 프로그램 |
| 요청 (Request) | 클라이언트가 서버에 "이거 줘/이거 해줘" 보내는 것 |
| 응답 (Response) | 서버가 클라이언트한테 돌려주는 결과 |
| HTTP | 요청/응답을 주고받을 때 쓰는 규칙(프로토콜) |
| URL | 자원의 주소. `https://도메인/경로?쿼리` 형태 |
| 포트 (Port) | 같은 컴퓨터에서 여러 서버 구분하는 번호. 로컬 개발 시 `localhost:8000` 처럼 씀 |

### HTTP 메서드 (Week4~5에서 실습)

| 메서드 | 의미 | 예시 |
|---|---|---|
| GET | 데이터 조회 | 게시글 목록 가져오기 |
| POST | 데이터 생성 | 새 게시글 작성 |
| PUT/PATCH | 데이터 수정 | 게시글 수정 |
| DELETE | 데이터 삭제 | 게시글 삭제 |

### 상태 코드 (일부만)

| 코드 | 의미 |
|---|---|
| 200 | 성공 |
| 201 | 생성 성공 |
| 400 | 잘못된 요청 (클라이언트 잘못) |
| 401/403 | 인증/권한 없음 |
| 404 | 자원 없음 |
| 500 | 서버 내부 에러 |

## 3. 프론트엔드 3요소

| 언어 | 역할 | Python 대비 비유 |
|---|---|---|
| HTML | 뼈대/구조 (제목, 문단, 버튼 등 뭐가 있는지) | 데이터 구조 정의 |
| CSS | 스타일 (색, 크기, 배치) | 없음 (프론트 고유) |
| JavaScript | 동작 (클릭하면 반응, 데이터 가져오기) | Python이랑 문법 비슷한 프로그래밍 언어 |

- HTML은 "무엇이 있는지", CSS는 "어떻게 보이는지", JS는 "어떻게 동작하는지" 담당
- 셋은 역할이 완전히 분리되어 있음. HTML 안에 스타일/동작 억지로 넣지 않는 게 좋은 습관

## 4. 백엔드 기초 개념

| 용어 | 뜻 |
|---|---|
| 서버 프레임워크 | 서버 만들 때 반복 작업 대신 해주는 도구. 여기선 FastAPI |
| 라우팅 (Routing) | 어떤 URL 요청이 오면 어떤 코드를 실행할지 연결하는 것 |
| 엔드포인트 (Endpoint) | 서버가 제공하는 개별 URL 주소 (예: `/todos`, `/users/1`) |
| API | 프론트/다른 프로그램이 서버 기능을 쓸 수 있게 만든 창구. 엔드포인트 모음 |
| CRUD | Create/Read/Update/Delete. 데이터 다루는 4가지 기본 동작 |
| CORS | 다른 출처(도메인/포트)끼리 통신 허용하는 브라우저 보안 정책. 프론트-백엔드 포트 다를 때 설정 필요 (Week5) |

## 5. 데이터 형식: JSON

프론트-백엔드는 보통 JSON으로 데이터 주고받음. Python의 dict와 거의 1:1 대응.

```json
{
  "id": 1,
  "title": "장보기",
  "done": false,
  "tags": ["집안일", "긴급"]
}
```

```python
# Python dict로 보면 이렇게 생김
{
    "id": 1,
    "title": "장보기",
    "done": False,      # JSON은 false (소문자), Python은 False
    "tags": ["집안일", "긴급"],
}
```

차이점: JSON은 `true`/`false`/`null` (소문자), Python은 `True`/`False`/`None`.

## 6. DB 기초 개념

| 용어 | 뜻 |
|---|---|
| DB (Database) | 데이터를 구조적으로 저장하는 시스템 |
| 테이블 (Table) | 엑셀 시트 같은 것. 행(row)과 열(column)로 구성 |
| SQL | DB에 데이터 넣고 빼고 수정하는 명령어 언어 |
| ORM | SQL 직접 안 쓰고 Python 코드로 DB 다루게 해주는 도구 (여기선 SQLAlchemy) |

## 7. 개발 환경 용어

| 용어 | 뜻 |
|---|---|
| 터미널 / CLI | 명령어로 컴퓨터 조작하는 화면 |
| IDE / 에디터 | 코드 작성 프로그램 (VSCode) |
| 저장소 (Repository, repo) | Git으로 관리하는 프로젝트 폴더 |
| 커밋 (Commit) | 코드 변경사항의 저장 지점 |
| 브랜치 (Branch) | 독립적으로 작업할 수 있는 코드 흐름 갈래 |
| 배포 (Deploy) | 만든 걸 인터넷에서 남들도 접속 가능하게 올리는 것 |
| 패키지 / 라이브러리 | 남이 만들어둔 재사용 가능한 코드 모음 (Python의 pip 패키지, JS의 npm 패키지) |
| 가상환경 (venv) | 프로젝트별로 Python 패키지를 독립적으로 관리하는 공간 |
