## 2. 요구사항 정의

> 📝 **작성 가이드**
> "이 서비스가 뭘 할 수 있어야 하는가"를 목록으로 정리하는 섹션. 화면이나 DB보다 먼저 정해야 함 — 기능이 정해져야 화면/DB가 뒤따라옴.

### 2.1 기능 요구사항 (Functional Requirements)

> 사용자가 실제로 할 수 있는 행동을 동사로 나열. "~할 수 있다" 문장으로 써보면 쉬움.
> 예: "사용자는 지출 항목을 추가할 수 있다", "사용자는 카테고리별로 지출을 필터링할 수 있다"
>
> 우선순위 나누는 이유: 시간이 부족하면 Must만 완성해도 프로젝트가 동작해야 하기 때문.
>
> - **Must**: 이게 없으면 서비스라고 부를 수 없음 (핵심 CRUD 등)
> - **Should**: 있으면 훨씬 좋지만 없어도 서비스는 동작함
> - **Could**: 시간 남으면 추가하는 보너스 기능

| ID   | 기능 | 설명 | 우선순위 |
| ---- | ---- | ---- | -------- |
| F-01 |      |      | Must     |
| F-02 |      |      | Should   |
| F-03 |      |      | Could    |

### 2.2 비기능 요구사항 (Non-Functional Requirements)

> "무엇을 하는가"가 아니라 "어떻게 동작해야 하는가"를 적는 곳. 처음엔 감이 안 올 수 있음 — 아래 예시 참고해서 최소한만 채워도 충분.

- **성능**: (예: 목록 조회 1초 이내 응답)
- **보안**: (예: 비밀번호 평문 저장 금지, 입력값 검증)
- **접근성/반응형**: (예: 모바일 화면에서도 사용 가능)
- **배포 환경**: (예: GitHub Pages + Render 무료 플랜)

## 3. 화면 설계 (Screen Design)

> 📝 **작성 가이드**
> 2번에서 정한 기능들을 "사용자가 어느 화면에서 어떻게 쓰는지"로 구체화하는 섹션. 그림을 잘 그릴 필요 없음 — 어디에 뭐가 있는지만 명확하면 됨.

### 3.1 화면 목록

> 서비스에 필요한 화면(페이지)이 몇 개인지, 각각 뭘 하는 화면인지 나열. "관련 기능 ID"로 2.1의 기능과 연결하면 빠진 화면 없는지 확인 가능.

| 화면 ID | 화면명         | 설명 | 관련 기능 ID |
| ------- | -------------- | ---- | ------------ |
| S-01    | 메인/목록 화면 |      | F-01         |
| S-02    | 상세 화면      |      | F-02         |
| S-03    | 입력/수정 화면 |      | F-01         |

### 3.2 화면 흐름도 (User Flow)

> 사용자가 화면들을 어떤 순서로 이동하는지 화살표로 표현. "화면 A에서 버튼 누르면 화면 B로 이동" 식으로 3.1의 화면 ID를 연결하면 됨.

```
[진입] → [S-01 목록 화면] → (항목 클릭) → [S-02 상세 화면]
                          → (추가 버튼) → [S-03 입력 화면] → (저장) → [S-01로 복귀]
```

### 3.3 와이어프레임

> 실제 디자인이 아니라 "레이아웃 배치도". 어디에 헤더, 어디에 버튼, 어디에 목록이 있는지만 보여주면 됨.
> 방법은 자유: 종이에 손그림 그려서 사진 찍어 첨부 / 아래처럼 텍스트 박스로 표현 / Figma 사용 시 링크만 남겨도 됨.

화면별로 대략적인 레이아웃을 텍스트/그림/링크로 첨부. (손그림 사진, Figma 링크, ASCII 박스 다 가능)

```
[S-01 목록 화면 예시]
+--------------------------------+
| 헤더 (제목, 네비게이션)          |
+--------------------------------+
| [검색창]              [추가 +] |
+--------------------------------+
| 항목1                          |
| 항목2                          |
| 항목3                          |
+--------------------------------+
| 푸터                            |
+--------------------------------+
```

### 3.4 화면별 상세 명세

> 와이어프레임의 각 요소(버튼, 입력창 등)가 눌렸을 때/입력됐을 때 무슨 일이 일어나는지 표로 정리. "이 버튼 누르면 어디로 가는지" 같은 동작을 적으면 나중에 구현할 때 헤매지 않음.

각 화면마다 아래 표 복사해서 작성.

**S-01. 화면명**

| 구성 요소 | 타입      | 설명                               |
| --------- | --------- | ---------------------------------- |
| 제목      | 텍스트    |                                    |
| 검색창    | input     | 실시간 필터링 or 버튼 클릭 시 검색 |
| 목록      | list/card | 항목 클릭 시 상세 이동             |
| 추가 버튼 | button    | 클릭 시 S-03 이동                  |

## 4. 데이터 모델 설계 (이 부분은 작성하지 않아도 됨)

> 📝 **작성 가이드 — DB를 처음 다루는 경우 꼭 읽기**
>
> **데이터 모델 = 우리 서비스가 저장해야 할 정보를 "엑셀 표"처럼 정리하는 것.** 어렵게 느껴지면 이렇게 생각하면 됨:
>
> - **테이블(table)** = 엑셀 파일의 시트 하나. 예: "지출 내역" 시트, "사용자" 시트
> - **컬럼(column)** = 시트의 열(세로줄) 제목. 예: 날짜, 금액, 카테고리
> - **행(row)** = 시트의 실제 데이터 한 줄. 예: `2026-07-13 | 5000원 | 식비`
> - **PK (Primary Key)** = 그 행을 구별하는 고유 번호. 엑셀에서 자동으로 매겨지는 줄 번호라고 생각하면 됨. 보통 `id` 컬럼 하나 만들고 자동 증가시킴.
> - **FK (Foreign Key)** = 다른 시트를 참조하는 컬럼. 예: "지출 내역" 시트에 "누가 썼는지"를 적어야 하면, `user_id` 컬럼에 "사용자" 시트의 id를 적어 넣는 것.
>
> **어떻게 만들면 되나 (순서)**
>
> 1. 2.1에서 적은 기능들을 보면서 "이 기능을 동작시키려면 어떤 정보를 저장해야 하나"를 나열 (예: 지출 추가 기능 → 날짜, 금액, 카테고리, 메모가 필요)
> 2. 비슷한 정보끼리 묶어서 테이블 하나로 만듦 (예: 지출 관련 정보는 전부 `expenses` 테이블에)
> 3. 테이블이 2개 이상이고 서로 관련 있으면 (예: 사용자마다 자기 지출만 보여야 함) FK로 연결
> 4. 각 컬럼의 타입만 대략 정하면 됨 — 글자는 `TEXT`, 숫자는 `INTEGER`(정수)/`REAL`(소수), 날짜는 `DATETIME`
>
> 처음엔 완벽할 필요 없음. Week6(DB 연동) 실습 진행하면서 같이 다듬을 예정이니, 지금은 "어떤 정보가 필요한지" 목록 뽑는 정도면 충분.

### 4.1 ERD (개체-관계도)

> 테이블끼리 어떻게 연결되는지 그림으로 표현. 테이블이 1개뿐이면 이 섹션은 생략 가능.
> `1 --- N` 은 "하나의 User가 여러 개의 Item을 가질 수 있다"는 뜻 (예: 사용자 1명이 지출 내역 여러 개 작성).

```
[User] 1 --- N [Item]
  id            id
  name          user_id (FK)
  ...           title
                ...
```

### 4.2 테이블 명세

> 4.1에서 정한 테이블마다 아래 표를 복사해서, 어떤 컬럼이 있는지 하나씩 적기. "제약조건"이 어려우면 비워둬도 되고, 최소한 PK인 `id` 컬럼만은 표시.

**테이블명: `items`**

| 컬럼명     | 타입     | 제약조건          | 설명        |
| ---------- | -------- | ----------------- | ----------- |
| id         | INTEGER  | PK, AUTOINCREMENT | 고유 식별자 |
| title      | TEXT     | NOT NULL          |             |
| created_at | DATETIME | DEFAULT NOW       | 생성일시    |

(테이블별로 복사해서 추가)

## 5. API 명세

> 📝 **작성 가이드**
> 프론트엔드와 백엔드가 서로 데이터를 주고받는 "약속"을 미리 정하는 섹션. Week4~5에서 배우는 fetch/FastAPI와 바로 연결되는 부분.
> 4번에서 정한 테이블 하나당 보통 "목록 조회/단건 조회/생성/수정/삭제" 5개 API가 한 세트로 나옴 — 아래 표가 그 기본 패턴.
>
> - **메서드**: GET(조회) / POST(생성) / PUT(수정) / DELETE(삭제)
> - **엔드포인트**: 요청을 보낼 주소 (URL 경로)
> - **요청/응답**: 프론트가 보내는 데이터 / 서버가 돌려주는 데이터 (JSON 형태로 대략 어떤 필드가 오가는지만 적으면 됨)
>   지금 당장 다 정확할 필요 없음 — Week5 FastAPI 실습 때 맞춰가며 수정 가능.

| 메서드 | 엔드포인트        | 설명      | 요청(Request)  | 응답(Response)       |
| ------ | ----------------- | --------- | -------------- | -------------------- |
| GET    | `/api/items`      | 목록 조회 | -              | `[{id, title, ...}]` |
| GET    | `/api/items/{id}` | 단건 조회 | -              | `{id, title, ...}`   |
| POST   | `/api/items`      | 생성      | `{title, ...}` | `{id, title, ...}`   |
| PUT    | `/api/items/{id}` | 수정      | `{title, ...}` | `{id, title, ...}`   |
| DELETE | `/api/items/{id}` | 삭제      | -              | `204 No Content`     |

- **에러 응답 공통 형식**: `{ "detail": "에러 메시지" }`
- **인증 방식** (해당 시): (예: 없음 / 세션 / JWT)
